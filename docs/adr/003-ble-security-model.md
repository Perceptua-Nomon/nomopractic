# ADR-003: BLE Security Model

> ⚠️ **SUPERSEDED — Do not implement.**
> This ADR describes application-layer AES-128-CCM encryption, HKDF key derivation, and a custom pairing ceremony that were fully replaced by [ADR-004: BLE Simplification — Native OS Pairing + NDJSON Relay](004-ble-simplification.md).
> OS-level Bluetooth passkey pairing (BlueZ Passkey Entry, link-layer AES-CCM) replaces all app-layer crypto.
> The `ble/session.rs` Rust module and `lib/ble-session.ts` TypeScript module described here have been deleted.
> Retain this document for historical reference only.

## Status

Superseded by [ADR-004](004-ble-simplification.md) and [ADR-005](005-wifi-soft-ap.md)

## Date

2026-04-15

## Context

BLE 4.2 on the Pi Zero 2W (BCM43436s) supports "Just Works" pairing, which
provides encrypted transport but **no authentication** — any nearby device can
pair and send commands. Numeric Comparison pairing requires a display (the Pi
has none). Passkey Entry requires a keyboard on both sides.

nomon robots have physical actuators (motors, servos) controlled over BLE.
Unauthenticated command injection could cause the robot to drive into
obstacles or people. The security model must:

1. Authenticate the connecting client (prove it knows the pairing secret).
2. Protect command integrity (prevent tampering).
3. Protect command confidentiality (prevent eavesdropping on the session).
4. Detect and reject replay attacks.

## Decision

Implement **application-layer authentication and encryption** over BLE GATT,
independent of BLE-level pairing.

### Pairing Flow

```
┌──────────┐                              ┌──────────────┐
│nomotactic│                              │ nomopractic  │
│ (app)    │                              │ (Pi, GATT)   │
└────┬─────┘                              └──────┬───────┘
     │                                           │
     │ 1. Write pairing_secret to                │
     │    Pairing Secret characteristic          │
     │──────────────────────────────────────────▶│
     │                                           │ 2. Verify secret:
     │                                           │    hmac.compare_digest()
     │                                           │    Single-use consumption
     │                                           │
     │                                           │ 3. Generate:
     │                                           │    - 16-byte random salt
     │                                           │    - session_key = HKDF-SHA256(
     │                                           │        ikm=pairing_secret,
     │                                           │        salt=salt,
     │                                           │        info="nomon-ble-session",
     │                                           │        len=16)
     │                                           │    - JWT (HS256, iss=nomon-device)
     │                                           │
     │ 4. Notify Auth Token:                     │
     │    salt (16 bytes) || jwt_bytes           │
     │◀──────────────────────────────────────────│
     │                                           │
     │ 5. App derives same session_key           │
     │    from pairing_secret + salt + HKDF      │
     │                                           │
     │ 6. Subsequent commands: AES-128-CCM       │
     │    encrypted with session_key             │
     │──────────────────────────────────────────▶│
     │                                           │
```

### Key Derivation

```
session_key = HKDF-SHA256(
    ikm  = pairing_secret (UTF-8 bytes),
    salt = random_salt (16 bytes, generated by Pi),
    info = "nomon-ble-session" (ASCII),
    len  = 16  (AES-128 key)
)
```

Both sides independently derive the same `session_key`. The salt is
transmitted in the clear as part of the Auth Token notification — this is
safe because the pairing secret (the IKM) is never transmitted and provides
the entropy.

### Session Encryption

All BLE command frames after pairing use AES-128-CCM:

- **Key**: `session_key` (16 bytes, derived above)
- **Nonce** (13 bytes): `"NM" || direction_u8 || counter_u16_LE || 0x00 × 8`
  - `direction_u8`: `0x00` client→server, `0x01` server→client
  - `counter_u16_LE`: Monotonic per-direction counter (prevents replay)
- **Tag length**: 4 bytes (truncated — sufficient for short-range BLE)
- **AAD**: Frame header bytes (`opcode || seq_nr || length`)

Counter-based nonces ensure:
- No nonce reuse (monotonic increment)
- Replay detection (server rejects any counter ≤ last seen)
- Direction isolation (separate counter spaces per direction)

### JWT Issuance

nomopractic issues a JWT during BLE pairing with the same claims structure
as nomothetic's device-mode auth:

```json
{
  "sub": "device-owner@local",
  "iss": "nomon-device",
  "exp": <now + 24h>,
  "iat": <now>
}
```

- Signed with the shared `NOMON_JWT_SECRET` (same secret used by nomothetic).
- The JWT is delivered over BLE but used for HTTPS after WiFi is established.
- nomopractic uses the `jsonwebtoken` crate for HS256 signing.

### Shared Secrets

nomopractic and nomothetic share two secrets for BLE/HTTP interop:

| Secret | Source | Used by |
|--------|--------|---------|
| Pairing secret | Generated by nomothetic at startup, written to `/var/lib/nomon/pairing_secret` | nomopractic reads for BLE verification; nomothetic uses for HTTP pairing |
| JWT secret | `NOMON_JWT_SECRET` env var | nomopractic signs BLE-issued JWTs; nomothetic validates them |

The shared pairing secret file is owned by `root:nomon` with mode `0640`.

## Alternatives Considered

### 1. BLE-level "Just Works" pairing only

- Provides encryption but no authentication.
- Any device in range can pair and send motor commands.
- **Rejected** — unacceptable for physical actuator control.

### 2. BLE Passkey Entry

- Requires displaying a 6-digit code on the Pi (no display) or pre-sharing
  a fixed passkey (weak, no single-use guarantee).
- **Rejected** — hardware constraint (no display on Pi).

### 3. HMAC authentication without encryption

- Protects integrity but not confidentiality.
- BLE commands (motor speeds, servo angles) are visible to passive sniffers.
- While not secrets, command patterns could reveal robot behaviour to
  nearby adversaries.
- **Rejected** — user explicitly requires session encryption.

### 4. Full TLS over BLE (L2CAP CoC + TLS)

- Provides authenticated encryption with well-tested protocol.
- L2CAP Connection-Oriented Channels require BLE 4.1+ (available on Pi).
- `bluer` supports L2CAP but it bypasses GATT — breaks the characteristic-
  based API model. Complex setup, poor mobile library support.
- **Rejected** — over-engineered, poor ecosystem support.

### 5. App-layer auth + AES-128-CCM encryption (chosen)

- Pairing secret provides mutual authentication (app proves knowledge).
- HKDF provides proper key derivation with domain separation.
- AES-128-CCM provides authenticated encryption with low overhead
  (4-byte tag, counter-based nonce).
- Counter-based replay protection.
- JWT bridges BLE pairing to HTTPS authentication.
- **Accepted.**

## Consequences

- New Rust dependencies: `jsonwebtoken`, `hkdf`, `sha2`, `aes`, `ccm`
  (all from the RustCrypto project, well-maintained, no-std compatible).
- nomotactic must implement the same HKDF + AES-128-CCM in TypeScript.
  `react-native-ble-plx` handles raw characteristic writes; crypto is done
  in JS via `crypto-js` or `@noble/ciphers`.
- The shared pairing secret file (`/var/lib/nomon/pairing_secret`) is a new
  cross-process coordination point between nomopractic and nomothetic.
- BLE session state (session key, counters, connected client) is held in
  memory by nomopractic; lost on daemon restart (client must re-pair).
- The JWT issued over BLE is valid for HTTPS — if the app later connects
  via WiFi, it can use the same token without re-pairing.
- Security checklist must be updated with BLE-specific items.
